# 代码注释工作文档

## 1. 目的

本文档用于统一 CFIE 范围内的代码注释写法，确保后续新增注释、重写注释与批量补注释工作遵循同一套标准。

本规范的核心目标不是“补充几句说明”，而是让读代码的人能够快速理解以下三个层面：

- 每一行代码或每一个关键语句在当前流程中的作用。
- 每一段代码承担的主功能是什么。
- 整个函数的执行主线是如何向前推进的。

## 2. 强制规则

### 2.1 注释语言

- 所有注释统一使用中文。
- 不允许在同一份代码中混用中文注释与英文注释作为主表达方式。

### 2.2 旧注释处理方式

- 对原代码中的旧注释，必须先删除，再按当前规范重新编写。
- 不允许直接沿用原有注释内容。
- 不允许仅对原有注释做直译。
- 不允许把原有注释与新注释进行拼接后继续使用。

### 2.3 注释覆盖要求

- 不能只写方法注释或函数注释后就结束。
- 不能只对少量代码做合并说明，而让大段核心逻辑保持无注释状态。
- 默认要求是：函数体内每一个可执行语句、变量定义语句、状态更新语句、返回前计算语句，都应有紧邻的单行注释。
- 这里的“紧邻”是指注释放在该语句正上方，优先一条注释对应一条语句。
- 不允许只在一个代码块开头写总说明，然后让块内大部分语句裸露。
- 不允许把“关键语句”理解为“只挑少数核心语句写注释”。
- 如果若干连续语句的语义完全一致，且逐句拆开反而降低可读性，才允许极少量合并说明。
- 即使允许合并说明，也只能用于同类、同目的、同形状变化的一小段连续语句，不能跨逻辑阶段合并。
- 不允许出现大段代码只有一句笼统注释的情况。

### 2.4 注释表达要求

- 注释必须说明当前语句为什么这样写，以及它在当前流程中的作用。
- 注释不能只重复代码表面的字面意思。
- 注释必须帮助读者建立执行顺序、状态变化与上下游关系的理解。
- 注释默认使用单行短注释，优先短句，不写成长段背景介绍。
- 除空行、花括号、预处理行、明显属于同一语句续行的换行外，函数体里的实际执行代码都应被覆盖到。
- 如果读者在顺着函数体向下读时，看到某一条实际执行语句上方没有注释，应默认视为未达到规范。

## 3. 分区注释格式

对于逻辑较长、分支较多、步骤较明显的函数，必须加入分区注释。分区注释要放在一个逻辑阶段开始之前，用来明确指出该阶段代码的核心职责。

分区注释固定格式如下：

```python
# ------------------------------- 该分区代码的主要功能 -------------------------------
```

分区注释应当准确描述该阶段在做什么，例如：

- 校验输入参数
- 初始化运行时状态
- 执行批量加载
- 计算中间结果
- 构造返回结果

分区注释的作用不是装饰排版，而是把一个较长函数拆解为多个可快速扫描的逻辑阶段，使读者能够先看到主线，再进入细节。

## 4. 行注释要求

普通行注释应紧贴代码语义，准确说明这条语句或这一小段语句在当前流程中的职责。

### 4.1 应当做到

- 说明当前语句处理的对象是什么。
- 说明当前语句为什么要在这里执行。
- 说明当前语句会影响什么状态、输出或后续分支。
- 在必要时指出该语句与前后文之间的衔接关系。
- 单行注释的每一行都应尽量精简，优先点明当前语句职责，不展开过多背景说明。
- 默认使用“短句 + 当前职责”的写法，不写泛泛的段落式解释。
- 一条注释优先只服务于一条语句，避免一条注释同时覆盖多条不同职责的代码。

### 4.2 不应当这样写

- 不能把代码直接换一种说法再写一遍。
- 不能只写“这里做初始化”“这里做处理”这类过于空泛的说明。
- 不能省略关键状态变化、条件判断目的或异常分支含义。
- 不能把“这一段是在做某某”当作整段几十行代码的唯一注释。
- 不能只在函数开头解释主线，而不解释函数体内部每条实际执行语句。

### 4.3 合并说明的边界

- 只有在多行语句语义完全一致、并且读者不会因为合并说明而失去理解时，才允许合并注释。
- 只要语句之间存在明显的职责差异、状态变化差异或顺序依赖，就应拆开解释。
- 变量定义、张量变换、状态回写、索引构造、条件分支、返回值拼装，原则上都应逐句解释。

### 4.4 张量与形状注释要求

- 只要语句涉及 tensor 计算、tensor 变换、tensor 索引、tensor 广播、tensor 拼接、tensor 回写，就应在注释中标明主要张量形状。
- 形状标注应尽量使用代码中真实出现的维度名，例如 `[B, T, H, D]`、`[num_tokens, hidden]`、`[T_chunk, V]`。
- 如果语句前后形状发生变化，注释应明确写出变化前后，例如 `[B, T, Hg, K] -> [B, T, H, K]`。
- 对 `view`、`reshape`、`permute`、`transpose`、`unsqueeze`、`squeeze`、`flatten`、`repeat_interleave`、`cat`、`stack`、`gather`、`scatter`、`index_select`、`slice`、`select`、`unfold`、`matmul`、`softmax`、`cumsum`、`copy_` 等常见张量操作，不允许省略形状说明。
- 对 `B`、`T`、`H`、`K`、`V`、`N`、`num_reqs`、`seq_len`、`chunk_size` 这类维度或流程核心元数据定义，也应补一条简短注释说明含义。
- 如果语句只处理标量，但这个标量决定了张量布局、索引范围或状态推进，也应在注释里说明它服务于哪个形状或哪个阶段。

### 4.5 推荐写法示例

推荐：

```cpp
// q_chunk: [T_chunk, K]
auto q_chunk = q_all.select(1, head_idx);

// attn_chunk: [T_chunk, T_chunk]
auto attn_chunk = torch::matmul(q_chunk, k_chunk.transpose(0, 1));

// 把当前 chunk 输出回写到 out[batch_idx, start:end, head_idx, :]
out[batch_idx].slice(0, start, end).select(1, head_idx).copy_(out_chunk);
```

不推荐：

```cpp
// 这里做 attention 计算
auto attn_chunk = torch::matmul(q_chunk, k_chunk.transpose(0, 1));

// 这里写回结果
out[batch_idx].slice(0, start, end).select(1, head_idx).copy_(out_chunk);
```

## 5. 函数注释要求

函数级注释用于说明函数整体职责与执行主线，但函数注释不能替代行注释与分区注释。

编写函数相关注释时，应确保读者能够从注释中明确看出：

- 函数的主要输入是什么。
- 函数的主要输出是什么。
- 函数按什么顺序推进关键步骤。
- 哪些阶段负责校验、准备、执行、收尾或返回。

如果函数本身较短且逻辑单一，可以减少分区数量；但只要函数已经形成明显阶段，就必须补充分区注释。

需要特别注意：

- 函数注释、分区注释只能帮助读者理解主线，不能替代函数体内部逐语句单行注释。
- 判断注释是否合格时，应以“函数体逐行向下阅读时，每条实际执行语句是否都能被立刻解释”为准。

## 6. 整体风格要求

所有注释在整体风格上必须满足以下要求：

- 术语统一：同一概念在同一文件中应使用统一称呼，不随意切换表述。
- 语义精确：注释必须准确描述真实行为，不能模糊、夸大或省略关键条件。
- 层次清晰：函数主线、分区职责与行级语义应形成清楚的阅读层次。
- 可直接使用：注释应达到可直接替换进代码并投入使用的质量，不依赖二次润色。

## 7. 验收标准

完成注释后，应至少满足以下检查标准：

- 读者能够快速看出函数整体执行主线。
- 读者能够快速理解每个逻辑阶段分别在做什么。
- 函数体内每条实际执行语句原则上都有对应的简短单行注释。
- 所有 tensor 计算、tensor 变换、tensor 索引与 tensor 回写语句，都标明了核心形状。
- 不存在长段核心逻辑裸露无注释，也不存在只注释函数头或只注释少量“关键语句”的情况。
- 注释内容与代码真实行为一致，不存在照抄旧注释、直译旧注释或空泛复述代码的问题。
- 注释整体读起来术语统一、语义准确、层次稳定，可以直接作为代码中的正式注释使用。
