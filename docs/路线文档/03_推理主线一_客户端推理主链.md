# 推理主线一：客户端推理主链

## 1. 文档定位

本文件定义 CFIE 推理侧的主线一，也就是当前客户端推理主链。

它替代原 `CFIE/docs/inference_overview.md` 的主链说明，用于统一维护推理器的目标、主请求链、模型加载链、MoE tiered cache、配置体系与当前状态。

## 2. 主线目标

主线一回答的问题是：

- 如何在客户端 / 桌面端 / 低资源环境中，把 CFIE 做成一条可运行的推理主链。

当前统一目标包括：

- 显存节省优先于计算便利。
- 内存节省优先于实现简单。
- `122B` 目标可落地优先于开发期 `35B` 运行舒适度。
- 参数按需搬运优先于全量常驻。
- MoE 专家支持 CPU / NVMe 分层驻留与按需装载。

## 3. 主链范围

主线一当前覆盖下面这些内容：

- CLI / Python SDK 入口。
- `EngineConfig` 组装与校验。
- request queue、scheduler、executor、worker、output processor 主链。
- Qwen3.5 MoE 模型加载链与量化落点。
- MoE tiered cache / 动态专家加载。
- 配置体系与运行时预算。
- 分布式 / DP / wave coordination 的当前实现。
- vLLM 依赖边界与吸收现状。

它不负责 predictor 专用 candidate routing 接入；那属于推理主线二。

## 4. 当前主链

当前 Phase 1 的最小推理路径可以概括为：

`CLI / Python SDK -> EngineConfig 校验 -> Request queue -> FCFS scheduler -> InputBuilder -> Executor(ModelRunner) -> OutputProcessor -> Streamer`

从 `LLMEngine` 视角，默认主请求链是：

`input_processor -> engine_core -> output_processor`

当前默认约束是：

- 先以单进程 / 单设备路径为主。
- 默认 FCFS。
- 当前主链优先面向 `max_num_seqs=1` 场景。
- 当前阶段明确把 `MoE offload / tiered cache` 视作 MoE 模型的原生默认显存压力卸载方式。
- 对 Qwen3.5 MoE，只要文档、命令或 profile 没有特别声明替代方案，都默认按 `tiered cache + expert unload/on-demand reload` 理解运行主链。
- `UVA`、`prefetch` 与其他通用 CPU offload 通路，暂不作为 Qwen3.5 MoE 主链成立前提，也不是当前 MoE 主线的先行验收项。
- `UVA`、`prefetch` 与其他通用 CPU offload 通路，后续优先用于非 MoE 模型推理稳定化；等非 MoE 推理路径稳定后，再单独推进这些通道的通用化。

## 5. 当前模型与加载链

当前核心模型选型是 Qwen3.5 MoE 系列，设计目标需要面向 `122B`。

当前文档里明确过的关键事实包括：

- 推理顶层实际模型对象不是 dense 版本，而是 `Qwen3_5MoeForConditionalGeneration`。
- 权重加载主链会经过 `GPUModelRunner.load_model()`、`initialize_model(...)`、`DefaultModelLoader.load_weights()` 与 `Qwen3_5Model.load_weights()`。
- `Qwen3_5Model.load_weights()` 是当前最关键的手工映射入口。
- routed experts 是 GPTQ Int4 的核心承载区域，动态排除规则通常会绕开 `attn.*`、`shared_expert.*`、`mtp.*`、`visual.*`。

## 6. MoE tiered cache 与动态专家加载

主线一当前的一个核心特点是：

- GPU 常驻的不是全量 experts，而是少量 routed expert slots。
- 其余 experts 落到 CPU 静态缓存或 NVMe safetensors。
- 推理时根据 router 选中的 experts，把缺失 expert 动态装入 GPU slot。

这条链的意义在于：

- prefill 阶段不会一次把 256 个专家全搬进 GPU。
- GPU 侧常驻的是 slot，不是全量专家。
- CPU / NVMe 侧承担了大多数 routed expert 的冷存储与回源职责。
- 对 Qwen3.5 MoE 推理，专家内存卸载默认开启；如果命令、profile 或文档未单独声明关闭，都按 `tiered cache + expert unload/on-demand reload` 理解。
- 当前阶段文档约束是：Qwen3.5 MoE 的显存压力优先通过 `tiered cache` 与专家按需回载解决；`MoE offload` 是当前原生默认方案，而不是把 `UVA` 当成 MoE 主链的默认前提。
- `UVA`、`prefetch` 与其他通用 CPU offload 路径，当前主要面向后续非 MoE 推理场景做稳定化，不应倒逼当前 MoE 主链先依赖这些能力才能成立。

## 7. 关键命令与入口

当前推理主线一最关键的命令入口如下：

- `serve`
  - 轻量本地引擎入口，适合主链验证。
- `run-local`
  - 本地运行别名入口。
- `chat` / `native-chat`
  - 交互式推理入口。
- `native-generate`
  - 更贴近原生 v1 执行链的单次请求入口，适合验证底层主链。

## 8. 当前状态

当前结论可以压缩为一句话：

- CFIE 已经不是简单套壳 vLLM 的脚本集合，而是在形成自己的客户端推理引擎主链。

更具体地说：

- Qwen3.5 MoE 路径已经形成清晰的加载链、量化分支和 MoE tiered cache 动态加载结构。
- GPU 常驻的是少量 slot + 按需换入，而不是全量 experts。
- vLLM 依赖仍然很多，但已经进入“边界清楚、按链路切片吸收”的执行阶段。

## 9. 当前未完成项

主线一后续仍需继续收口的重点包括：

- 继续围绕低显存、低内存与按需搬运做资源收口。
- 继续梳理 vLLM 依赖边界，推进按链路内生化。
- 继续稳住 `MoE offload / tiered cache`、KV cache、线性注意力层、MTP 与 MoE 动态执行。
- 在非 MoE 模型推理范围内，再单独推进 `UVA` / `prefetch` 等通用 CPU offload 通路的稳定化。
- 在通用 offload 通路没有独立稳定之前，继续把 `MoE offload / tiered cache` 作为 MoE 推理的默认产线通道维护。

## 10. 主线约束补充

- `build_moe_tiered_cache_plan()` 优先复用正式权重加载链来解析 `model_path`。如果配置里给的是 Hugging Face repo id，会先经过 `prepare_weights(...)` 解析到本地模型目录，再读取 safetensors 元数据做规划。
- GPU 静态预算与 runtime headroom 统一基于“当前可用显存”切分，而不是基于物理总显存硬切。`gpu_memory_utilization` 仍然是同一条主线的分账比例，但它作用在 `available GPU memory` 上。
- 当前 Qwen3.5 MoE 主线把 CPU 侧定义为“全量专家静态镜像”。只要 tiered cache 启用，CPU 侧默认就要能容纳全部 routed experts，而不是只容纳 GPU 未常驻的那一部分。
- 当前主线不再把 NVMe 当作 Qwen3.5 MoE 推理的默认二级缓存，也不再把 staging buffer 当作主链成立前提。动态换入专家的默认方向是 `CPU static mirror -> GPU resident slot`。
- `staging_bytes` 与 `nvme_expert_bytes` 当前主要保留给旧 plan 字段兼容；主线规划默认把它们收敛为 `0`。
- CPU 预算口径已经简化为两类约束：`available system memory - shared reserve` 与显式配置的 `moe_cpu_budget_gb` 硬上限；`moe_cpu_min_free_gb` 不再作为主线规划的决定性约束。
- 因此，当前推理主线一可以概括为：`可用显存切静态预算 -> GPU 至少保住 top-k 常驻 -> CPU 全量专家镜像 -> 运行时只做 CPU -> GPU 动态换入`。
